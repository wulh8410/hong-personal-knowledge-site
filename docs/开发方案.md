# 个人主页 / 微信生态知识库官网开发方案

> 项目用途：个人品牌展示、日常文章更新、微信生态知识库沉淀、SEO/GEO 收录、AI 搜索可识别  
> 适用对象：Codex / AI 编程助手 / 前端开发者  
> 建议技术栈：Next.js + TypeScript + Tailwind CSS + MDX/Markdown + 静态生成 + Vercel 部署  
> 当前版本：V1.0  
> 后台策略：第一阶段不做复杂后台，先用 Markdown 文件管理内容；第二阶段再接 Decap CMS 或其他轻量 CMS。

---

## 0. 项目一句话说明

开发一个“个人品牌 + 微信生态知识库 + GEO 内容资产站”。

网站不做成传统个人简历站，也不做成强销售型官网，而是做成一个长期沉淀内容、展示专业能力、方便搜索引擎和 AI 工具理解的知识库型个人主页。

核心目标：

1. 展示个人在微信生态电商、微信小店、微信推客、小程序商城、微信豆投放、AI 工具与 SEO/GEO 方面的专业能力。
2. 支持持续发布文章、专题知识库、案例与工具资源。
3. 让搜索引擎和 AI 搜索工具更容易理解网站主体、专业领域、内容结构和作者身份。
4. 初期不做复杂后台，优先用 Markdown/MDX 管理内容，降低开发与维护成本。
5. 后续可平滑升级为带后台的内容管理系统。

## 1. 项目定位

### 1.1 网站定位

网站名称暂定：

**宏的微信生态实战笔记**

也可以根据最终品牌调整为：

- 宏记
- 宏的微信生态知识库
- 微信生态与 AI 实战笔记
- 微信小店 / 推客 / AI 实战知识库

### 1.2 主定位

**微信生态电商与 AI 实战知识库**

### 1.3 副定位

长期深耕微信小店、微信推客、小程序商城、微信豆投放与 AI 工具应用，持续整理微信生态规则、产品能力、实战经验和内容增长方法。

### 1.4 用户访问后的核心认知

用户访问网站后，要能快速理解：

1. 站长是谁。
2. 站长擅长什么。
3. 站长长期研究哪些领域。
4. 网站有哪些可持续更新的知识库。
5. 哪些文章值得优先阅读。
6. 如何联系作者或进一步合作。

### 1.5 网站不做什么

第一版先不做：

- 复杂会员系统
- 复杂后台 CMS
- 在线支付
- 课程售卖
- 社区论坛
- 评论系统
- AI 问答助手
- 复杂 CRM
- 视频课程播放系统

这些功能后续可以扩展，但 V1 不做。

## 3. 总体开发原则

### 3.1 第一阶段原则

V1 阶段遵循：

- 内容优先
- 结构优先
- SEO/GEO 优先
- 快速上线
- 少后台
- 少动态数据库
- 高可维护
- 静态生成优先
- 移动端适配优先

### 3.2 技术路线

推荐：

```txt
Next.js App Router
TypeScript
Tailwind CSS
MDX / Markdown
Static Site Generation
GitHub
Vercel
```

### 3.3 为什么第一版不用复杂后台

第一版网站的核心不是做后台，而是把内容结构和站点框架跑起来。

文章、知识库、案例、工具资源全部通过 Markdown/MDX 文件管理。

优点：

- 开发速度快
- 不需要数据库
- 不需要登录权限系统
- 部署简单
- SEO 友好
- 适合 Codex 辅助维护
- 后续容易升级 CMS

用于描述作者实体。

字段：

```json

  "@context": "https://schema.org",

  "name": "宏",

  "url": "https://your-domain.com",
  "sameAs": []
}
```

#### WebSite

用于描述网站。

字段：

```json
{
  "@context": "https://schema.org",
  "@type": "WebSite",
  "name": "宏的微信生态实战笔记",
  "url": "https://your-domain.com",
  "description": "微信小店、微信推客、微信豆投放、小程序商城、AI 工具与 SEO/GEO 内容资产知识库。"

```

#### Article

用于文章页。

#### BreadcrumbList

用于知识库、文章、案例页面。

#### FAQPage

如果文章 frontmatter 有 faq 字段，则生成 FAQPage。

## 21. 第一版不做后台，但要预留后台扩展

### 21.1 当前内容管理方式

第一阶段：

```txt
通过 /content 目录中的 Markdown/MDX 文件管理内容。
```

新增文章流程：

```txt
1. 在 /content/articles 新建 .md 或 .mdx 文件
2. 填写 frontmatter
3. 编写正文
4. 提交 GitHub
5. Vercel 自动部署
```

### 21.2 后续后台方案

第二阶段可接：

```txt
Decap CMS
```

目标：

- 网页后台新增文章
- 网页后台编辑文章
- 上传封面图
- 管理分类和标签
- 仍然保存为 Markdown 文件
- 自动提交到 GitHub

### 21.3 预留要求

开发时请保证：

- 内容读取逻辑集中在 `lib/content.ts`
- 不要把文章内容写死在组件里
- 不要把知识库配置写死在多个页面中
- 所有内容数据尽量从 Markdown 或配置文件读取
- 后续接 CMS 时，只需要替换内容来源，不需要重构页面
